## Introduction to Containerization

### The "It Works on My Machine" Problem

Historically, developers faced significant friction when moving code from a local laptop to a production server. Differences in operating systems, library versions, and system configurations often caused applications to fail. Containerization solves this by bundling the application code together with its dependencies, libraries, and configuration files into a single unit.

### Containers vs. Virtual Machines

While both provide isolation, they do so differently:

- **Virtual Machines (VMs):** Each VM includes a full guest operating system, making them bulky and slow to start.
- **Containers:** Containers share the host system's kernel and isolate only the application processes. This makes them lightweight, fast, and highly portable.

## Docker Core Concepts

### Key Terminology

- **Image:** A read-only template containing the instructions for creating a Docker container.
- **Container:** A runnable instance of an image. You can start, stop, move, or delete a container using the Docker API or CLI.
- **Registry:** A storage system for Docker images (e.g., Docker Hub).
- **Volume:** A mechanism for persisting data generated by and used by Docker containers.

### Defining the Recipe: The Dockerfile

A Dockerfile is a text document that contains all the commands a user could call on the command line to assemble an image.

### Common Dockerfile Instructions

- **FROM:** Sets the base image for subsequent instructions (e.g., `python:3.9-slim`).
- **WORKDIR:** Sets the working directory for any following instructions.
- **COPY:** Copies new files or directories from your local machine into the container.
- **RUN:** Executes commands in a new layer on top of the current image and commits the results (e.g., `pip install`).
- **CMD:** Provides defaults for an executing container. There can only be one CMD instruction in a Dockerfile.

### Example: Packaging a Flask App

Below is a standard Dockerfile setup for a portable Flask application:

```dockerfile
# Use an official Python runtime as a parent image
FROM python:3.9-slim

# Set the working directory in the container
WORKDIR /app

# Copy the current directory contents into the container at /app
COPY . /app

# Install any needed packages specified in requirements.txt
RUN pip install --no-cache-dir -r requirements.txt

# Make port 5000 available to the world outside this container
EXPOSE 5000

# Define environment variable
ENV FLASK_APP=app.py

# Run app.py when the container launches
CMD ["flask", "run", "--host=0.0.0.0"]
```

### Basic Commands

Once your Dockerfile is ready, you interact with it using the Docker CLI:

```bash
# Build an image from a Dockerfile in the current directory
docker build -t my-flask-app .

# Run a container based on that image
docker run -p 5000:5000 my-flask-app

# List running containers
docker ps
```
